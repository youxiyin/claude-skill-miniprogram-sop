# P0 · 需求澄清（喂给 agent 的指令）

你要从用户的**一段话描述**，自动起草一个新小程序的领域知识资产。**这一步不写任何代码。**

## 输入
用户的一段话（贴在本指令后面）。

## 你要做的（按序）
1. **判品类**：医疗健康 / 电商批发 / 社交内容 / 工具效率 / 预约服务 / 其它。判不准就选最接近的 + 记一句理由。
2. **套红线库**：读 `redline-library/<品类>.md` + `redline-library/_generic.md`，作为风险登记册的**预置底稿**。
3. **抽结构化规格**：
   - 核心用户是谁、最想完成的事；
   - **3–5 条核心闭环**（用户从进入到达成目标的完整链路）；
   - **页面清单**（每页：叫什么、放什么、能去哪）；
   - 数据实体（存什么、在哪存：本地 / 云 / 都要）。
4. **起草三份资产**（用 `templates/` 的空模板填，输出到项目根）：
   - `spec-checklist.md`：每条能力 → UI 入口 → 期望落点 → 怎么验（端到端可达且符合意图）。
   - `risk-register.md`：红线库预置项 + **从描述推断的新增项**，逐条标 严重度/触发条件/守护(待补锁测试)/状态=❌未护。
   - `user-journeys.md`：核心闭环 J1…Jn，每步"应发生什么"。
5. **只在关键信息缺失且无法安全默认时**，提 **2–3 个澄清问题**（挑信息价值最高的，打包问，别零敲碎打）。品类必问项：
   - 医疗：是否会**替医生下判断**？（红线：只帮不漏事、不替判断）
   - 电商：是否涉**真实支付/金额结算**？
   - 社交/内容：是否有 **UGC 需内容安全审核**？是否触未成年？
   - 任何品类：**核心数据丢了用户会不会受害**？（决定持久化强度）
   - 凡能安全默认的（如次要配色、非关键文案）**别问**，带默认 + 标注假设写进 spec。

## 输出
- 三份资产草稿（`spec-checklist.md` / `risk-register.md` / `user-journeys.md`）；
- 品类判定 + 一句理由；
- （若有）澄清问题清单。

## 交接闸（人闸①）
有澄清问题 → 等用户答；无 → 带"已标注假设的默认"直接进 P1。**假设写进 spec，后续可被推翻。**

## 铁律（本阶段相关）
- 领域知识**你起草、人审批**——别让用户填空。
- 红线**宁多勿漏**：模板里的红线除非明显不适用，一律保留为❌未护，交给循环去护。
- **不替人做产品取舍**：遇"该不该有某功能"这类主观题 → 记进 spec 的"待定"，不自己拍死。
