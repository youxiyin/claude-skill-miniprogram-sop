# 规格清单 spec-checklist（P0 起草，agent 填 · 人闸①审）

判一条能力"完成"的标准 = **端到端可达 + 行为符合用户意图**，不是"代码里有这函数"。

## 能力矩阵
| 编号 | 能力（用户想做的事） | UI 入口（从哪点进） | 期望落点（点了应到哪/发生什么） | 怎么验（E2E/旅程/单测） | 状态 |
|---|---|---|---|---|---|
| SPEC-1 | 〈填〉 | 〈哪页哪个按钮〉 | 〈落点页面/数据变化〉 | 〈验证方式〉 | ❌未做 |
| SPEC-2 | | | | | ❌未做 |

## 假设与待定（带默认、可被推翻）
- 假设A：〈P0 没问用户、带的默认 + 理由〉
- 待定B：〈涉及产品取舍、不自己拍，留给人闸④〉

## 覆盖状态
- `status`: DRAFT / PARTIALLY-COVERED / FULLY-COVERED（除 human 项）
- 循环每 `spec_coverage_every` 轮对照本清单找"该有却走不到/落点错"的负空间 bug。
